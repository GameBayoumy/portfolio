# 📋 Versioning System Best Practices Guide\n\n## 🎯 Overview\n\nThis guide provides comprehensive best practices for using the versioning system in the XR Developer Portfolio. Follow these guidelines to ensure consistent, reliable, and automated version management.\n\n## 🔄 Workflow Guidelines\n\n### 1. Daily Development Workflow\n\n```bash\n# 1. Start your work day\ngit checkout main\ngit pull origin main\n\n# 2. Create feature branch\ngit checkout -b feat/linkedin-timeline-enhancement\n\n# 3. Make your changes with conventional commits\ngit add .\ngit commit -m \"feat(linkedin): add interactive timeline hover effects\"\n\n# 4. Push and create PR\ngit push -u origin feat/linkedin-timeline-enhancement\ngh pr create --title \"feat(linkedin): add interactive timeline hover effects\" --body \"Implements hover effects for better UX\"\n\n# 5. After PR approval and merge, automatic release happens\n# Version is automatically bumped and deployed\n```\n\n### 2. Commit Message Standards\n\n**✅ Good Examples:**\n```bash\nfeat(linkedin): add professional timeline component\nfix(github): resolve API rate limiting issue\ndocs: update installation instructions  \nperf(3d): optimize model loading performance\nrefactor(api): consolidate service layer\n```\n\n**❌ Bad Examples:**\n```bash\nFixed bug                          # No type, no description\nAdd new feature                    # Vague, no scope\nUPDATE: LinkedIn stuff             # Wrong case, unclear\nfeat: stuff                        # Too vague\n```\n\n### 3. Semantic Versioning Rules\n\n| Change Type | Version Bump | Example |\n|-------------|-------------|----------|\n| 🚨 **Breaking Changes** | MAJOR (1.0.0 → 2.0.0) | `BREAKING CHANGE: remove deprecated API` |\n| ✨ **New Features** | MINOR (1.0.0 → 1.1.0) | `feat(linkedin): add skills radar chart` |\n| 🐛 **Bug Fixes** | PATCH (1.0.0 → 1.0.1) | `fix(github): resolve contribution display` |\n| 📚 **Documentation** | PATCH (1.0.0 → 1.0.1) | `docs: update API documentation` |\n| ⚡ **Performance** | PATCH (1.0.0 → 1.0.1) | `perf(3d): reduce bundle size by 20%` |\n\n## 📦 Release Management\n\n### Automated Releases (Recommended)\n\nReleases happen automatically when you push to `main`:\n\n```bash\n# Push to main triggers:\n# 1. Commit validation\n# 2. Tests and linting\n# 3. Version calculation\n# 4. Changelog generation\n# 5. Git tag creation\n# 6. GitHub release\n# 7. Vercel deployment\n```\n\n### Manual Releases (When Needed)\n\n```bash\n# Dry run to see what would happen\nnpm run release:dry\n\n# Create patch release\nnpm run release:patch\n\n# Create minor release\nnpm run release:minor\n\n# Create major release (use carefully!)\nnpm run release:major\n\n# Create beta release\nnpm run release:beta\n```\n\n### Emergency Procedures\n\n**🚨 Emergency Rollback:**\n```bash\n# Via GitHub Actions\ngh workflow run release.yml --field release_type=rollback\n\n# Or manually\nnode scripts/vercel-deploy-hook.js rollback\n```\n\n**🔍 Release Validation:**\n```bash\n# Check version info\nnpm run version:info\n\n# Validate deployment\ncurl -I https://your-domain.vercel.app/api/version\n\n# Run performance check\nnpm run lighthouse:local\n```\n\n## 🎯 Portfolio-Specific Guidelines\n\n### LinkedIn Integration Updates\n\n```bash\n# LinkedIn feature additions\nfeat(linkedin): add endorsements visualization\n# Results in: \"Commit message validation failed\"**\n```bash\n# Solution: Fix commit message format\ngit commit --amend -m \"feat(scope): proper description\"\n```\n\n**❌ \"Vercel deployment failed\"**\n```bash\n# Solution: Check build logs and environment variables\nnpm run build  # Test locally first\nnpm run version:check  # Validate dependencies\n```\n\n**❌ \"Performance checks failing\"**\n```bash\n# Solution: Run performance analysis\nnpm run lighthouse:local\nnpm run analyze\n```\n\n### Debug Commands\n\n```bash\n# Check semantic release dry run\nnpm run release:dry\n\n# Validate commit
